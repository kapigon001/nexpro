# CS NRR最大化 3年ロードマップ

---

## Executive Summary

本ロードマップは、ネクプロCSチームがNRR（Net Revenue Retention）を3年間で段階的に向上させるための戦略・体制・基盤構築を定義する。

### 核心的課題

**解約理由の74%が「施策終了・中止」**

これはツールの問題ではなく、**顧客のウェビナーマーケティング施策自体が成果を出せずに頓挫している**ことを意味する。つまり、CSが「ツールの使い方」を教えても解約は防げない。**顧客が成果を出せる状態を作る**ことが唯一の解である。

### 3年目標

| 指標 | Year 1 | Year 2 | Year 3 |
|------|--------|--------|--------|
| **NRR** | 105% | 115% | 125% |
| **Gross Churn Rate** | 12% | 8% | 5% |
| **Expansion Rate** | 17% | 23% | 30% |

### 戦略の柱

1. **成果定義の標準化** - 顧客ごとに「何をもって成果とするか」を明確化し、測定可能にする
2. **先回り介入の仕組み化** - 利用状況・成果データから危険信号を検知し、顧客が諦める前に介入
3. **成功顧客からの拡大** - 成果が出ている顧客への横展開・アップセル提案を仕組み化
4. **顧客データ基盤の構築** - Salesforceを中心に、利用状況・成果・活動履歴を一元管理

## 1. NRR最大化の基本戦略

### NRRの構造

```
NRR = 100% - Gross Churn + Expansion

例：NRR 115% = 100% - 8%(解約) + 23%(拡大)
```

NRRを上げるには「解約を減らす（守り）」と「拡大を増やす（攻め）」の両輪が必要。

### 1.1 Gross Churn低減（守り）

#### 解約の根本原因への対処

| 解約理由 | 割合 | 根本原因 | 対策 |
|----------|------|----------|------|
| 施策終了・中止 | 74% | 成果が出ない/見えない → 費用対効果が合わないと判断 | 成果定義の標準化、ROI可視化、施策改善提案 |
| その他 | 26% | 担当者変更、予算削減、競合乗換等 | 複数キーパーソン構築、早期検知・介入 |

#### 解約防止の3層構造

```
┌─────────────────────────────────────────────┐
│ 第1層：成果創出支援（根本解決）             │
│ → 成果定義、測定、改善提案                  │
├─────────────────────────────────────────────┤
│ 第2層：早期検知・介入（予防）               │
│ → ヘルススコア監視、Yellow段階で介入        │
├─────────────────────────────────────────────┤
│ 第3層：解約阻止（最終防衛）                 │
│ → Red顧客への緊急対応、条件交渉             │
└─────────────────────────────────────────────┘
```

### 1.2 Expansion拡大（攻め）

#### Expansionの種類

| 種類 | 内容 | トリガー |
|------|------|----------|
| **アップセル** | 上位プラン、追加ライセンス | 利用量が上限80%到達、成果目標達成 |
| **クロスセル** | 追加機能、オプション | 特定機能の頻繁利用、新たな課題の発見 |
| **横展開** | 他部署への導入 | 導入部署での成功、組織内チャンピオン育成 |
| **紹介** | 他社への紹介 | NPS高スコア、成功事例化の許諾 |

#### Expansion創出の原則

**「成果が出ている顧客からしかExpansionは生まれない」**

したがって、Expansion戦略は「成果創出支援」の延長線上にある。成果を出す → 成功を認識させる → 次のステップを提案する、という流れを設計する。

## 3. 顧客マスタ管理基盤（Salesforce集約型）

### 3.1 現状の課題

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│ Salesforce  │     │   board     │     │  ネクプロ   │
│ (企業/商談) │     │ (請求/売上) │     │ (利用状況)  │
└──────┬──────┘     └──────┬──────┘     └──────┬──────┘
       │                   │                   │
       │    連携なし       │    連携なし       │
       ▼                   ▼                   ▼
┌─────────────────────────────────────────────────────────┐
│                     手動で突合・集計                     │
│              （スプシ、Excel、属人的管理）               │
└─────────────────────────────────────────────────────────┘
```

**問題点：**
- 顧客の全体像を把握するのに複数システムを見る必要がある
- 担当引き継ぎ時に情報が散逸する
- 利用状況がCSに見えないため、先回り対応ができない
- 成果・ROIの蓄積場所がない

### 3.2 目指す姿

```
                    ┌─────────────────────────────────┐
                    │         Salesforce              │
                    │      （Single Source of Truth） │
                    ├─────────────────────────────────┤
                    │ ・企業情報（アカウント）        │
                    │ ・商談/契約情報                 │
                    │ ・活動履歴（議事録、メール）    │
                    │ ・利用状況（自動連携）          │
                    │ ・成果/ROI情報                  │
                    │ ・ヘルススコア                  │
                    └────────────┬────────────────────┘
                                 │
        ┌────────────────────────┼────────────────────────┐
        │                        │                        │
        ▼                        ▼                        ▼
┌───────────────┐      ┌───────────────┐      ┌───────────────┐
│    board      │      │   ネクプロ    │      │    Slack      │
│  (請求連携)   │      │ (利用状況)    │      │  (アラート)   │
└───────────────┘      └───────────────┘      └───────────────┘
     API/CSV               API連携              Webhook
```

### 3.4 手動更新を最小化する仕組み

#### 自動化できる領域

| データ | 連携元 | 連携方式 | 頻度 |
|--------|--------|----------|------|
| 利用状況 | ネクプロDB | API連携（後述） | 日次 |
| 売上/MRR | board | CSV自動インポート or API | 月次 |
| ヘルススコア | Salesforce内 | 数式/フロー自動計算 | リアルタイム |
| アラート | Salesforce | フロー→Slack通知 | リアルタイム |

#### 手動入力が必要な領域（効率化の工夫）

| データ | 入力者 | 効率化の工夫 |
|--------|--------|-------------|
| 議事録 | CSM | 議事録テンプレート、音声文字起こし→整形 |
| 成果目標 | CSM | オンボーディング時に成果定義シートで必ず設定 |
| 成果実績 | CSM | 月次QBR/レビュー時にまとめて入力 |
| キーパーソン情報 | CSM | 面談後に更新（入力項目を最小限に） |

#### 入力負荷軽減のルール

1. **必須項目は最小限に** - 入力率を上げるため、本当に必要な項目のみ必須化
2. **選択肢を用意** - 自由記述より選択式（ピックリスト）を優先
3. **入力タイミングを明確化** - 「面談後24時間以内」「月末」など
4. **モバイル入力対応** - 移動中でも入力できるようSalesforceモバイル活用

### 3.5 利用状況の自動連携方法

#### 推奨アーキテクチャ

```
┌─────────────────┐
│   ネクプロDB    │
│ (PostgreSQL等)  │
└────────┬────────┘
         │ 日次バッチ
         ▼
┌─────────────────┐
│   ETL/連携層    │
│ (選択肢は下記)  │
└────────┬────────┘
         │ Salesforce API
         ▼
┌─────────────────┐
│   Salesforce    │
│ 利用状況__c     │
└─────────────────┘
```

#### 連携方式の選択肢

| 方式 | 概要 | メリット | デメリット | 推奨フェーズ |
|------|------|----------|-----------|-------------|
| **A. CSV手動インポート** | 週次でCSVエクスポート→データローダーでインポート | 即日開始可能、コストゼロ | 手動作業、タイムラグ | Year1 Q1-Q2 |
| **B. Salesforce Connect** | 外部DBをSalesforceから直接参照 | リアルタイム、開発少 | 参照のみ（書込不可）、ライセンス費 | Year1 Q3〜 |
| **C. Heroku Connect** | Heroku経由でPostgreSQL⇔Salesforce同期 | 双方向同期、柔軟 | Heroku費用、構築工数 | Year2〜 |
| **D. カスタムAPI連携** | 自社開発でAPI連携バッチ構築 | 完全カスタマイズ | 開発工数、保守コスト | Year2〜 |
| **E. iPaaS（Tray.io等）** | ノーコードで連携フロー構築 | 開発不要、柔軟 | 月額費用 | Year1 Q3〜 |

#### 推奨ロードマップ

```
Year1 Q1-Q2：CSV手動インポート（週次）
    ↓
Year1 Q3-Q4：iPaaS（Tray.io/Make）で日次自動化
    ↓
Year2〜：Heroku Connect or カスタムAPI連携でリアルタイム化
```

### 4.2 KPI定義

#### NRR関連（最重要）

| KPI | 定義 | 計算式 |
|-----|------|--------|
| NRR | 既存顧客からの売上維持・拡大率 | (期末MRR + Expansion - Churn) / 期首MRR |
| Gross Churn Rate | 解約による売上減少率 | 解約MRR / 期首MRR |
| Expansion Rate | 既存顧客からの売上拡大率 | (アップセル + クロスセル) MRR / 期首MRR |

#### 先行指標（Churn予測）

| KPI | 定義 | 目標 |
|-----|------|------|
| ヘルススコア「Green」比率 | 健全な顧客の割合 | 70% |
| ヘルススコア「Red」比率 | 危険な顧客の割合 | 5%未満 |
| オンボーディング完了率 | 定義した完了基準を満たした顧客の割合 | 90% |
| Time to First Value | 契約から初回成功体験までの日数 | 30日以内 |

#### 先行指標（Expansion予測）

| KPI | 定義 | 目標 |
|-----|------|------|
| 成果達成顧客比率 | 設定した成果目標を達成した顧客の割合 | 60% |
| QBR実施率 | Tier1/2顧客へのQBR実施率 | 80% |
| Expansion提案数 | 月間のアップセル/クロスセル提案件数 | CSM1人あたり月5件 |
| 提案→成約率 | 提案からの成約率 | 30% |

### C. 用語集

|------|------|
| NRR（Net Revenue Retention） | 既存顧客からの売上維持・拡大率。100%超で成長、100%未満で縮小 |
